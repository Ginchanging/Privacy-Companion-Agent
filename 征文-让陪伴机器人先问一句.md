# 让陪伴机器人先问一句：我们在 DGX Spark 上做出的"隐私优先"Demo

最初的构想很直接：做一个能感知用户回家、听懂用户说今天有点累，并主动推荐音乐或调节环境的陪伴助手。但讨论中团队提出一个问题：若系统长期记录作息、独处状态和情绪变化，积累下来的就不再是几条交互记录，而是一份完整的生活画像。项目方向由此调整：先约束系统能做什么，再讨论它能做多好。

我们把 Step3-VL 限定为只输出结构化的状态假设和建议理由，不具备授权与执行权限，最终决定权收归确定性策略。音乐与空调分别生成独立的 action_id，用户同意播放音乐，不代表同意开启空调。公网请求只经独立的 external-connector 出口，原始图像、音频与完整对话均不离开本地，控制台会展示实际发出的 JSON，作为可核验依据。

正式开发前，团队先对 DGX Spark 进行只读审计，确认现场没有可用摄像头，也没有获得授权的真实影像，因此没有采用虚构实时链路，而是使用两张标注为合成场景的图像，调用真实的 Step3-VL 模型。测试环节反复出现问题：严格枚举校验一度将合法 JSON 拒之门外，经多轮修正，Python 自动化测试数量从 54 个扩展到 255 个，前端另有 32 个测试。

部署到 DGX 后，PowerShell 兼容性、镜像构建上下文、SSH 隧道等问题接连出现。第一次远端测试有两项未通过，团队没有将其记作成功，修复后才替换线上服务。StepAudio 一度连续超时，排查后确认并非模型故障，而是推理耗时平均约26秒，界面因此调整为文字优先显示、语音随后生成。

前端界面也经历多轮精简：最初信息密度过高，形似工程仪表盘，团队通过反复截图评审，逐步收敛为对话为主、审计为辅的结构。协作模式为：Codex 负责代码实现与测试执行，团队成员判断边界是否清晰、结果是否属实。

截至提交，视觉输入仍为合成插画而非实拍，空调控制仍为模拟执行；在线音乐已支持可选 Audius 预览，但真实可用性仍受公开曲目和网络条件影响，因此保留仓库内原创音频降级。我们理解的可信赖 AI，并非功能完备，而是在执行动作前，愿意先确认这是否确为用户本人的意愿。
